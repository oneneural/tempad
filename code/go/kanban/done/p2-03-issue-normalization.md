# T-P203: Issue normalization (Linear → domain.Issue)

**Ticket:** T-P203
**Phase:** 2 — Tracker Client (Linear)
**Status:** 🔲 TODO
**Spec:** Section 13.3, 4.1.1
**Deps:** T-P201, T-P101

## Problem

Linear's API returns raw GraphQL payloads. Need to normalize into clean `domain.Issue` structs.

## Solution

`normalizeIssue()` function that maps Linear fields to domain fields with proper type handling.

## Files

- `internal/tracker/linear/normalize.go`

## Work Items

- [ ] `normalizeIssue(raw) domain.Issue` with field mapping:
  - `labels` → lowercase all names
  - `blocked_by` → derive from inverse relations where type is `blocks`
  - `priority` → int only, non-int → nil
  - `assignee` → user ID or email
  - `created_at`, `updated_at` → parse ISO-8601
  - `branch_name` → from `branchName`
- [ ] Handle nil/missing fields gracefully (no panics)
- [ ] Unit tests with real Linear response fixture

## Acceptance Criteria

- [ ] `["Bug", "Frontend"]` → `["bug", "frontend"]`
- [ ] Priority `2` → `2`, priority `"high"` → nil
- [ ] Relation `{type: "blocks", relatedIssue: {…}}` → `blocked_by` entry
- [ ] Missing fields don't panic
- [ ] Unit tests pass

## Research Notes (2026-03-08)

- Linear's `priority` field is always numeric (0-4: No priority, Urgent, High, Medium, Low). The spec's concern about non-int priority values may be unnecessary — validate against real responses.
- `branchName` is auto-generated by Linear and may be null if the issue hasn't been branched. Handle nil gracefully.
- Label names from Linear are already stored as-is — lowercase normalization is our own convention for consistent matching.
